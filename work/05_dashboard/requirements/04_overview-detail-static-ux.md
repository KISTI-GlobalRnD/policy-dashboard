# 정책-기술 집중도 대시보드 및 상세 페이지 설계도

## 목적

이 문서는 `work/05_dashboard` 정적 산출물을 아래 3단 구조로 다시 정의하는 기준 문서다.

1. `index.html`: 전체를 일괄로 보는 정책-기술 집중도 대시보드
2. `detail-tech.html`: 하나의 기술 분야를 중심으로 보는 상세 페이지
3. `detail-policy.html`: 하나의 정책 보고서를 중심으로 보는 상세 페이지

핵심 목적은 아래와 같다.

1. 전체보기에서 각 정책이 어느 기술에 집중하고 있는지 바로 읽히게 한다.
2. 기술 중심 상세에서 해당 기술에 어떤 정책들이 모이는지 설명한다.
3. 정책 중심 상세에서 해당 정책이 어떤 기술로 분산·집중되는지 설명한다.
4. 대표 근거는 상세에서 확인하되, 전체보기는 구조와 집중도를 먼저 읽게 한다.

즉 이번 산출물은 `단일 보고서`가 아니라 `대시보드 + 2종 상세` 구조다.

## 왜 방향을 다시 바꾸는가

이전 설계는 아래 점에서 요구와 어긋났다.

- 전체를 한 번에 보는 기능보다 개별 브리핑 전달에 치우쳤다.
- `정책 x 기술` 관계보다 `기술별 카드`가 먼저 보였다.
- 상세를 없애면서 기술 중심/정책 중심 drill-down이 사라졌다.
- 데이터가 3개 기술 수준일 때는 성립해도, 현재처럼 활성 기술이 크게 늘어난 상태를 받치지 못한다.

2026-03-17 기준 현재 payload는 이미 `13개 활성 기술`과 다수의 교차 정책 분포를 가진다.

대표 예시:

- `AI-바이오 국가전략`은 `인공지능`, `소재`, `양자`, `첨단바이오`에 걸친다.
- `123대 국정과제`는 `에너지`, `이차전지`, `국방`, `사이버보안`, `우주항공`, `해양` 등 다수 기술에 분산된다.
- `정부 AX사업 전주기 원스톱 지원방안`은 현재 `인공지능`에 집중된다.

따라서 첫 화면은 `브리핑 카드`보다 `정책-기술 집중도 맵`이어야 한다.

## 전달 우선순위

### A. 전체보기가 답해야 하는 질문

1. 어떤 정책이 어떤 기술에 집중하는가
2. 어떤 기술이 어떤 정책들로 구성되는가
3. 어디가 단일 기술 집중이고, 어디가 다기술 분산인가

### B. 기술 상세가 답해야 하는 질문

1. 이 기술 분야에 연결된 정책은 무엇인가
2. 그 정책 아래 어떤 주요 과제가 있는가
3. 대표 근거는 무엇인가

### C. 정책 상세가 답해야 하는 질문

1. 이 정책은 어떤 기술들에 연결되는가
2. 어느 기술에 가장 강하게 집중하는가
3. 기술별로 어떤 주요 과제가 매핑되는가

## 정보 구조

### 1. 전체보기 `index.html`

역할:

- 정책-기술 관계를 한 번에 보는 메인 대시보드

핵심 구성:

1. 상단 제목줄
2. 전체 집계
3. 정책 x 기술 집중도 매트릭스
4. 보조 랭킹/요약
5. 상세 진입 링크

레이아웃 골격:

- 상단 1행: 제목 + 전체 집계
- 본문 주영역: 정책 x 기술 매트릭스
- 본문 보조영역: 기술별 연결 정책 수, 정책별 상위 집중 기술

즉 overview의 주연은 매트릭스다. 카드형 브리핑 블록이 매트릭스를 밀어내면 안 된다.

#### A. 상단 제목줄

포함 항목:

- 문서 제목
- 활성 기술 수
- 정책 수
- 주요 과제 수
- 세부 조치 수

규칙:

- 구조 설명문 금지
- `정적 페이지`, `브리핑 보드` 같은 표현 금지
- 가장 먼저 `정책이 기술에 어떻게 분포하는가`가 읽혀야 한다

#### B. 정책 x 기술 집중도 매트릭스

축 정의:

- 행: 정책 보고서
- 열: 활성 기술 분야

셀 정의:

- 1차 수치: `group_count`
- 2차 수치: `content_count`
- 색 농도: `group_count` 기준
- 셀 표기 형식: `3 / 5`
  - 앞 숫자 = 주요 과제 수
  - 뒤 숫자 = 세부 조치 수

현재 데이터에 맞는 예시:

- `AI-바이오 국가전략 × 인공지능`: 3개 과제 / 5개 세부 조치
- `AI-바이오 국가전략 × 첨단바이오`: 1개 과제 / 2개 세부 조치
- `정부 AX사업 전주기 원스톱 지원방안 × 인공지능`: 3개 과제 / 6개 세부 조치
- `AI반도체 산업 도약 전략 × 반도체디스플레이`: 3개 과제 / 6개 세부 조치

규칙:

- `0건 기술`은 열에 노출하지 않는다.
- 기술 열은 `display_order`를 따른다.
- 정책 행은 전체 `group_count` 내림차순, tie 시 정책명 순으로 정렬한다.
- 셀 숫자는 `과제 수 / 세부 조치 수`를 같이 보인다.
- `0 / 0` 상태는 빈 셀 또는 약한 톤으로 처리한다.
- 각 정책 행에서 최대값 셀 1개는 강조한다.
- 각 기술 열에서도 상위 집중 정책 1~2개는 시각적으로 구분한다.

행/열 헤더 규칙:

- 행 헤더 본문: 정책명
- 행 헤더 보조 수치: 정책 전체 주요 과제 수
- 열 헤더 본문: 기술명
- 열 헤더 보조 수치: 연결 정책 수

범례 규칙:

- 매트릭스 위 또는 아래에 범례 1줄만 둔다.
- 예: `셀 숫자 = 주요 과제 수 / 세부 조치 수`
- 예: `진한 색일수록 주요 과제 연결이 많음`

#### C. 보조 요약

매트릭스 아래 또는 오른쪽에 아래 2개를 둔다.

1. `기술별 연결 정책 수`
2. `정책별 상위 집중 기술`

이 블록의 목적은 사용자가 매트릭스를 읽기 전에 패턴을 빠르게 파악하게 하는 것이다.

보조 요약 세부 규칙:

- `기술별 연결 정책 수`
  - 기술명
  - 연결 정책 수
  - 주요 과제 수
- `정책별 상위 집중 기술`
  - 정책명
  - 상위 1~3개 기술
  - 각 기술의 주요 과제 수

현재 데이터 기준 예시:

- `123대 국정과제 → 우주항공 3, 해양 3, 국방 2`
- `AI-바이오 국가전략 → 인공지능 3, 첨단바이오 1, 소재 1`
- `AI반도체 산업 도약 전략 → 반도체디스플레이 3`

#### D. 상세 진입

- 정책명 클릭 → `detail-policy.html#<policy-id-or-slug>`
- 기술명 클릭 → `detail-tech.html#<tech-domain-id>`
- 셀 클릭 → 기본적으로 정책 상세의 해당 기술 anchor로 이동

클릭 규칙:

- 정책 행 헤더 클릭은 `정책 중심 상세`의 진입점이다.
- 기술 열 헤더 클릭은 `기술 중심 상세`의 진입점이다.
- 셀 클릭은 `정책 중심 상세` 내 해당 기술 섹션으로 보내는 것이 기본값이다.
- 1차 구현에서는 셀 클릭을 생략하고 행/열 헤더 클릭만 두는 것도 허용한다.

### 1-1. overview에 이번 단계에서 넣지 않는 것

- 대표 근거 전문 노출
- 과제별 카드 펼침
- 상세 페이지 미리보기
- 내부 식별자
- 검토 상태
- raw item trace

즉 overview는 `판단 보드`로 유지하고, 추적 정보는 다음 단계 상세로 넘긴다.

## 기술 상세 `detail-tech.html`

주의:

- 이 섹션은 다음 단계 확정 대상이다.
- 이번 턴에서는 overview drill-down의 도착점 수준으로만 유지한다.

역할:

- 하나의 기술 분야를 중심으로 연결 정책과 과제를 읽는 페이지

페이지 구성:

1. 기술 헤더
2. 요약 수치
3. 연결 정책 개요
4. 정책별 주요 과제 보드
5. 대표 근거
6. 출처

### A. 기술 헤더

포함 항목:

- 기술명
- 한 줄 해석
- 연결 정책 수
- 주요 과제 수
- 세부 조치 수

### B. 연결 정책 개요

정책 카드마다 아래를 둔다.

- 정책명
- 이 기술에 연결된 과제 수
- 이 기술에 연결된 세부 조치 수
- 지원 유형 분포

### C. 정책별 주요 과제 보드

각 정책 아래에 해당 기술과 연결된 `groups[]`만 배치한다.

카드 항목:

- 주요 과제명
- 지원 유형
- 세부 기술
- 세부 조치 수

### D. 대표 근거

기술 상세에서는 아래 우선순위로 1~2개만 전면 노출한다.

1. 기술 내 최대 `content_count`를 가진 과제의 대표 근거
2. 필요 시 같은 기술의 다른 정책에서 보강 근거 1개

규칙:

- `원문 세부 항목`, `source_asset_id`, `review_required`는 본문에 두지 않는다.
- 문서명, 위치의 humanized label, 발행일만 남긴다.

## 정책 상세 `detail-policy.html`

주의:

- 이 섹션도 다음 단계 확정 대상이다.
- 이번 턴에서는 overview에서 어떤 정책 상세로 들어갈지 정의하는 수준만 유지한다.

역할:

- 하나의 정책 보고서가 어떤 기술들에 연결되는지 읽는 페이지

페이지 구성:

1. 정책 헤더
2. 기술 분포 요약
3. 기술별 주요 과제 보드
4. 대표 근거
5. 출처

### A. 정책 헤더

포함 항목:

- 정책명
- 연결 기술 수
- 주요 과제 수
- 세부 조치 수
- 한 줄 해석

한 줄 해석 예시:

- `AI-바이오 국가전략은 인공지능에 가장 강하게 연결되며, 첨단바이오·소재·양자까지 확장된다.`

### B. 기술 분포 요약

기술별로 아래를 표시한다.

- 과제 수
- 세부 조치 수
- 해당 정책 내 집중도 순위

### C. 기술별 주요 과제 보드

각 기술 섹션 아래에 해당 정책과 연결된 주요 과제만 배치한다.

카드 항목:

- 과제명
- 지원 유형
- 세부 기술
- 세부 조치 수

### D. 대표 근거

정책 상세에서는 해당 정책 안에서 가장 큰 기술 클러스터의 대표 근거를 1차로 둔다.
필요 시 두 번째 기술 클러스터의 보강 근거 1개를 붙인다.

## 데이터 매핑 기준

### 전체보기

- 기술 열: 활성 `tech_domains[]`
- 정책 행: `tech_domains[].groups[].policy`
- 셀 집계:
  - 과제 수: policy x tech 별 `groups[]` 개수
  - 세부 조치 수: 해당 groups의 `contents[]` 합계

### 기술 상세

- 페이지 단위: `tech_domains[n]`
- 정책 카드: 해당 기술 아래 고유 `policy`
- 과제 카드: `tech_domains[n].groups[]`
- 대표 근거: 포커스 `group.contents[].primary_policy_evidence`

### 정책 상세

- 페이지 단위: payload 전체를 훑어 특정 `policy_name`으로 재집계
- 기술 분포: policy x tech 집계 결과
- 과제 카드: policy_name이 일치하는 모든 group
- 대표 근거: policy 내 최대 tech cluster의 포커스 content

## 시각 원칙

- overview는 `대시보드`, detail은 `브리핑`처럼 읽혀야 한다.
- overview의 중심은 매트릭스다. 카드가 주연이 되면 안 된다.
- detail의 중심은 기술/정책 축 설명과 대표 근거다.
- 13개 안팎의 활성 기술이 들어와도 overview가 무너지지 않게 설계한다.
- 0건 기술은 보여주지 않는다.

## 검증 기준

아래 조건을 모두 만족해야 한다.

1. overview에서 정책 x 기술 관계가 첫 화면에 보인다.
2. overview만 보고도 각 정책의 상위 집중 기술을 읽을 수 있다.
3. tech detail에서 연결 정책들이 정책별로 분리되어 보인다.
4. policy detail에서 연결 기술들이 기술별로 분리되어 보인다.
5. 본문에 내부 식별자와 QA 메타가 보이지 않는다.
6. 0건 기술이 overview/detail 어디에도 나오지 않는다.
7. 대표 근거는 detail에서만 전면 노출되고, overview는 구조 읽기를 우선한다.
