# QA 체크리스트 v0.1

## 0. 범위 및 등록부 QA

- [ ] 모든 분석 대상 문서가 `04_document-registry.csv`에 등록되어 있다
- [ ] 각 문서의 역할이 `policy_source`, `benchmark_source`, `taxonomy_source`, `working_note`, `context_note` 중 하나로 지정되어 있다
- [ ] 포함 여부가 `include`, `hold`, `support`, `missing` 중 하나로 표시되어 있다
- [ ] 원문 미확보 문서는 `missing`으로 표시되어 있다
- [ ] 범위 이슈가 의사결정 로그에 기록되어 있다

## 1. 추출 QA

- [ ] 모든 추출 결과에 `document_id`가 있다
- [ ] 텍스트 블록은 페이지 또는 시트 기준으로 역추적 가능하다
- [ ] 표는 셀 구조를 보존한다
- [ ] 표 후보가 `structured`, `layout_false_positive`, `visual_table`, `multi_page_fragment`, `review_required` 중 하나로 분류된다
- [ ] 문서별 표 검토 큐가 존재한다
- [ ] 그림은 캡션 또는 요약 메타데이터를 가진다
- [ ] 추출 신뢰도 또는 품질 등급이 남아 있다
- [ ] 실패한 문서는 누락이 아니라 실패 상태로 기록되어 있다

## 2. 정규화 및 매핑 QA

- [ ] 문서명, 기관명, 날짜가 표준화되어 있다
- [ ] 정책항목은 최소 1개 이상의 근거 객체와 연결된다
- [ ] 전략 매핑은 대표 전략을 1개 이상 가진다
- [ ] 기술분야 매핑은 대분류까지는 최대한 채워져 있다
- [ ] 자원유형 분류 기준이 일관되다
- [ ] `unmapped` 또는 `review_required` 항목이 별도로 관리된다

## 3. 온톨로지 QA

- [ ] ID 체계가 중복 없이 유지된다
- [ ] 핵심 엔터티와 관계가 최소 데이터모델과 일치한다
- [ ] provenance 필드가 누락되지 않았다
- [ ] canonical table은 `ready`, `needs_normalization`, `needs_merge` 중 하나의 상태를 가진다
- [ ] 관계형 구현과 시맨틱 확장 포인트가 구분되어 있다

## 4. 대시보드 QA

- [ ] 모든 수치가 원문 근거로 내려갈 수 있다
- [ ] 전략, 기술분야, 자원유형 필터가 일관되게 동작한다
- [ ] 문서 상세에서 원문 페이지 또는 근거 객체로 이동 가능하다
- [ ] 보류 문서와 지원 문서가 본 분석 수치에 혼입되지 않는다

## 5. 통합 QA

- [ ] 샘플 문서 기준 end-to-end 흐름이 재현 가능하다
- [ ] 데이터 계약과 실제 산출물이 일치한다
- [ ] 누락 데이터와 보류 데이터가 명시적으로 구분된다
- [ ] 주요 의사결정 변경 시 영향 범위가 기록된다
