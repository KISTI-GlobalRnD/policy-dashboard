# 텍스트 정규화 전략 v0.2

## 목적

정책 문서 원문을 대시보드 연결 전 단계에서 바로 쓸 수 있는 문단 단위 텍스트로 정리한다.

## 현재 적용 범위

- 텍스트형 PDF
- 구조형 HWPX
- 구형 HWP rough text
- 현재 핵심 샘플: `DOC-POL-006`, `DOC-POL-003`
- phase1 정책 문서 기준으로는 현재 direct 대상이 PDF 1건과 HWPX 10건이다.

## 처리 규칙

## 현재 구현 기준

- 본문 재구성의 주 소스: `PyMuPDF` bbox text block
- 표 보존의 주 소스: `PyMuPDF4LLM` page chunk Markdown table

텍스트형 PDF는 본문을 bbox block으로 다시 세우고 표는 별도 블록으로 유지한다.

- 구조형 HWPX / HWP의 주 소스: 원본 블록 추출 결과
- HWPX는 문단 순서를 최대한 그대로 유지한다.
- HWP는 rough text의 표 셀 잔재를 후처리 규칙으로 줄인다.

### 1. 페이지 푸터 제거

- `- 1`, `- 2` 같은 쪽번호 라인은 제거한다.
- bbox 위치와 정규식 패턴을 함께 사용한다.

### 2. 표 bbox 제외

- 추출된 표 bbox와 크게 겹치는 텍스트 블록은 본문 재구성에서 제외한다.
- 이렇게 해야 표 셀 조각이 pure text에 섞이지 않는다.

### 3. 표 Markdown 보존

- `|`로 시작하는 연속 행은 표 블록으로 본다.
- 표 블록은 일반 문단으로 풀지 않고 Markdown 표 형태를 유지한다.

### 4. 텍스트 block 내부 재분할

- 같은 raw block 안에서도 `-`, `**`, `***`, `ㅇ`, `□`, `※`, `3.` 같은 시작 표지가 나오면 새 단위로 분리한다.
- 이 단계로 한 페이지 안의 불릿 목록이 더 안정적으로 쪼개진다.

### 5. 줄바꿈 접합

- 같은 텍스트 단위 안에서 줄바꿈된 라인은 공백 기준으로 접합한다.
- 괄호나 따옴표 앞뒤는 과도한 공백을 만들지 않도록 단순 규칙으로 붙인다.
- 같은 줄 안에서 bbox가 겹치는 조각은 `x1` 정렬과 병합 규칙으로 다시 붙인다.
- 같은 문장이 여러 bbox block으로 쪼개진 경우는 `세로 간격`, `좌우 정렬`, `행 폭`을 함께 보고 다시 붙인다.

### 6. 짧은 연속 블록 병합

- 다음 블록이 새 제목/불릿/주석이 아니고 길이가 짧으면 이전 블록의 연속 문장으로 병합한다.
- 이 규칙으로 line wrap에 의해 끊긴 문장을 다시 잇는다.
- 가운데 정렬된 짧은 제목 조각은 cover/title line으로 보고 추가 병합한다.
- `② 신규 ...` 같은 원형 숫자 소제목은 heading으로 승격해 앞 문장과 붙지 않도록 막는다.

### 6-1. PDF 후처리 정리

- page 1 cover metadata 중 회의명, 날짜, 부처합동 표기는 pure text 본문에서 제거한다.
- 같은 page에서 반복되는 `현장의 목소리` heading은 첫 heading만 남긴다.
- PDF의 `*`, `**`, `***` 계열 보조설명은 bullet이 아니라 note로 분류한다.
- `* *`, `** *`처럼 분리된 footnote marker는 `**`, `***`로 정규화한다.
- `□`, `ㅇ`, `▪`, `※`, `-` 뒤 공백이 빠진 경우는 low-risk 범위에서 표기 간격을 보정한다.
- `table_markdown` 블록은 surface spacing 보정에서 제외하고 원문 추출 상태를 유지한다.
- `TBL-*.json`이 지워졌거나 없는 경우에는 `pages.json.tables[].bbox`를 fallback으로 사용해 표 overlap 제외를 유지한다.
- 본문/불릿/노트에는 exact phrase 치환과 짧은 조사 단위 fragment 치환을 함께 적용한다.
- fragment 치환 뒤 exact phrase 치환을 한 번 더 적용해서 연쇄 보정이 끝까지 반영되게 한다.
- 현재 `DOC-POL-006`에는 `전문가의견 등을 통해 선정`, `관계부처 및 유관협회 및 기관`, `AI 도구 구독비 지원 신설`, `콘텐츠 제작지원 확대` 같은 반복 표현 보정이 반영돼 있다.
- 추가로 `정책지원 방안 마련`, `금융지원 방안 마련`, `ICT 기반 의료시스템 해외진출 기반 조성`, `현지 인수․진출 병원` 같은 정책 문구도 exact 치환 대상으로 관리한다.
- `③(글로벌 진출 지원)`, `❷(정책 패키지)`처럼 문장 중간에 붙은 원형 숫자 소제목은 후처리에서 별도 블록으로 분리한다.

### 7. 구조형 문서 라벨 정리

- 문두 `공개`, `대외비` 같은 cover marker는 선두 heading이면 제거한다.
- HWP/HWPX 블록 중 `금 융`, `인 재`, `제 도`처럼 한 글자씩 띄어진 짧은 라벨은 `금융`, `인재`, `제도`로 정규화한다.
- `분야 1`, `참고`, `금융`, `인재`, `인프라`, `제도`처럼 다음 문단을 설명하는 짧은 라벨은 heading으로 승격한다.
- `전략`, `내용`처럼 연속으로 나타나는 짧은 셀 조각은 table-like run으로 보고 제거한다.
- `⦁`, `◦` 같은 불릿도 일반 bullet 표지로 취급한다.
- `*`, `**`로 시작하는 구조형 보조 설명 라인은 bullet이 아니라 note로 분류한다.
- 구조형 블록 내부에 줄바꿈이 남아 있으면, 다음 줄이 bullet/note/heading marker로 시작할 때는 별도 문단으로 분리한다.
- 같은 구조형 블록 안의 단순 줄바꿈은 공백으로 접합해서 embedded newline을 남기지 않는다.
- `ㅇ`, `□`, `☞`, `⇒` 같은 marker-only 조각은 noise로 제거한다.
- `* ①`, `* ▲`, `<`처럼 의미 없는 기호 조각은 noise로 제거한다.

## 현재 한계

- OCR 문서에는 그대로 적용하기 어렵다.
- 원문 자체의 띄어쓰기 손실과 줄바꿈 접합 후 생기는 어색한 공백은 일부 남는다.
- 표 Markdown 안의 HTML `<br>` 표식은 아직 추가 정리하지 않는다.
- heuristic merge라서 일부 페이지는 과병합 또는 미병합이 남을 수 있다.
- cover/header 성격 문구와 반복 citation은 아직 수동 검토가 필요하다.
- 구형 HWP는 표/도형이 본문과 완전히 분리되지 않으므로, 일부 라벨은 still rough 상태로 남을 수 있다.
- HWPX에서도 footnote 자체가 원문에서 깨져 있으면, `note` 분류 후에도 내용이 불완전한 줄이 남을 수 있다.
- PDF 본문 내부 띄어쓰기 손상은 여전히 rule-based 보정 한계가 있다.
- 프로젝트 특화 exact 치환이 늘어나면 다른 PDF로 일반화되지 않을 수 있으므로, 새 규칙은 반복 패턴이 확인된 뒤에만 추가한다.
- 후속 분류/표 검토 단계는 정규화 산출물을 읽으므로 병렬보다 순차 실행이 안전하다.
